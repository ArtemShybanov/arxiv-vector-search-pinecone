## 1. Опис проєкту

У цьому проєкті реалізовано повний пайплайн пошуку по наукових статтях arXiv: підготовка датасету, створення ембеддингів, завантаження в Pinecone, семантичний пошук, пошук з фільтрами, порівняння метрик схожості, chunking та гібридний пошук через BM25 + векторний пошук + Reciprocal Rank Fusion.

Структура репозиторію:

```text
.
├── .env
├── .gitignore
├── requirements.txt
├── data/
│   └── arxiv_subset.parquet
├── embeddings/
│   └── embeddings.npy
├── scripts/
│   ├── 01_prepare_data.py
│   ├── 02_embed.py
│   ├── 03_load_to_pinecone.py
│   ├── 04_search.py
│   ├── 05_chunking.py
│   └── 06_hybrid_search.py
└── README.md
```

`.env`, `data/` та `embeddings/` додано в `.gitignore`, щоб не комітити API-ключі, сирі дані та великі бінарні файли.

## 3. Частина 1 — Дані та вибір інструментів

### 3.1. Підготовка датасету

Дані беруться з arXiv metadata dataset. Скрипт `scripts/01_prepare_data.py` читає JSONL-файл, відбирає перші 10 000 валідних записів, залишає поля `id`, `title`, `abstract`, `authors`, `year`, `category` і зберігає результат у `data/arxiv_subset.parquet`.

Запуск:

```bash
python scripts/01_prepare_data.py
```

Очікуваний результат: у папці `data/` зʼявляється файл `arxiv_subset.parquet`, а в терміналі виводиться кількість статей, топ категорій, розподіл за роками та приклад першого запису.

### 3.2. Чому Pinecone, а не Qdrant або Chroma

Для цієї роботи обрано Pinecone, тому що це керована векторна база даних: не потрібно окремо піднімати сервер, налаштовувати storage, індекси, резервне копіювання або деплой. Для поточного завдання це зменшує кількість інфраструктурної роботи і дозволяє сфокусуватися на головному: ембеддинги, індексація, пошук, фільтри, chunking та hybrid search.

Qdrant є сильним варіантом для production-сценаріїв, особливо коли потрібен self-hosted deployment, більший контроль над інфраструктурою або open-source стек. Chroma зручна для локальних прототипів і простих RAG-проєктів, але для цього завдання Pinecone виглядає практичніше, бо він одразу дає cloud-hosted індекс, API, metadata filtering і просту інтеграцію з Python.

З точки зору ліцензії та розгортання маємо кілька альтернатив: Pinecone — managed cloud service, Qdrant можна використовувати як open-source/self-hosted або cloud, Chroma часто використовують локально або як легку embedded vector DB. В контексті цього завдання Pinecone підходить краще, бо не треба вручну адмініструвати базу, а serverless-індекс достатній для 10 000 документів.

### 3.3. Чому `allenai/specter2_base`, а не `all-MiniLM-L6-v2`

`all-MiniLM-L6-v2` - універсальна sentence-transformer модель для загальних текстів. Вона доцільна для типових коротких речень, FAQ, простого semantic search. Але в цій роботі корпус складається з наукових статей: title, abstract, автори, категорії arXiv. Для такого домену краще підходить SPECTER2, тому що ця колекція моделей створена саме для комплексних документів.

Модель `allenai/specter2_base` у model card описана як базова модель SPECTER2, яку можна використовувати для отримання ембеддингів на основі заголовку й анотації наукової статті або короткого текстового запиту. Тобто формат нашого інпуту `title + " [SEP] " + abstract` відповідає задачі пошуку наукових текстів краще, ніж універсальна MiniLM-модель.

Посилання на model card: https://huggingface.co/allenai/specter2_base

### 3.4. Рекомендована метрика схожості

Для цього індексу використовується `cosine`, тому що ембеддинги нормалізуються через `normalize_embeddings=True`. Для нормалізованих векторів cosine similarity і dot product дають однаковий порядок ранжування, бо довжина кожного вектора приблизно дорівнює 1. Це важливо при створенні індексу: метрика Pinecone повинна відповідати способу підготовки ембеддингів. Якщо ембеддинги нормалізовані, cosine є вибором by default.

## 4. Частина 2 — Отримання ембеддингів

Скрипт `scripts/02_embed.py`:

- нормалізує ембеддинги;
- зберігає результат у `embeddings/embeddings.npy`.

Запуск:

```bash
python scripts/02_embed.py
```

Очікуваний вивід:

```text
[INFO] Loaded rows: 10000
[INFO] Embeddings shape: (10000, 768)
[INFO] First embedding norm: близько 1.0000
[INFO] Saved embeddings to: embeddings/embeddings.npy
```

### Чому для нормалізованих ембеддингів cosine similarity еквівалентна dot product

Cosine similarity між двома векторами визначається як:

```text
cosine(a, b) = dot(a, b) / (||a|| * ||b||)
```

Якщо ембеддинги нормалізовані, то `||a|| = 1` і `||b|| = 1`. Тоді формула спрощується:

```text
cosine(a, b) = dot(a, b) / (1 * 1) = dot(a, b)
```

Тому для одиничних векторів cosine similarity і dot product дають однакові значення і однаковий порядок результатів.

### Чому `abstract` обрізається до 500 символів

Pinecone має обмеження на сумарний розмір metadata одного запису. Тому повний abstract краще зберігати локально у `parquet`-файлі, а в Pinecone класти тільки короткий фрагмент для швидкого перегляду результатів. Повний текст можна підтягувати за `arxiv_id` або локальним `paper_<номер>` після пошуку.

## 6. Частина 4 — Пошукові запити

Скрипт `scripts/04_search.py` виконує:

1. чистий семантичний пошук у Pinecone;
2. пошук з фільтрацією за `year` і `category`;
3. локальне порівняння трьох метрик:
   - cosine similarity;
   - dot product;
   - L2-distance.

Запуск:

```bash
python scripts/04_search.py
```

Тестовий запит:

```text
teaching machines to recognize objects in pictures
```

### Чи збігаються top-5 для cosine і dot product і чому

Так. У моєму експерименті результати повністю збіглися, що підтверджується виводом:
```
Cosine and dot top-5 are identical: True
```

Також збігаються не лише документи, а й значення score:

```
Cosine:
0.8294
0.8260
0.8254
...

Dot product:
0.8294
0.8260
0.8254
...
```
Причина в тому, що нормалізація (normalize_embeddings=True) робить довжину кожного вектора рівною 1, тому cosine similarity математично зводиться до dot product. Якщо всі вектори мають одиничну довжину, обидві метрики ранжують документи однаково.

### Чи відрізняються результати для L2 і чому
У моєму запуску результати для L2 також збіглися з cosine similarity:

```
Cosine and L2 top-5 are identical: True
```
Наприклад, перші п’ять документів для cosine similarity та L2-distance виявилися однаковими.

Для нормалізованих векторів L2-distance теж повʼязана з cosine similarity:

Це пояснюється тим, що для нормалізованих векторів існує прямий математичний зв’язок:

```text
||a - b||² = ||a||² + ||b||² - 2 * dot(a, b)
```

Оскільки після нормалізації: `||a|| = 1` і `||b|| = 1`, тоді:

```text
||a - b||² = 2 - 2 * dot(a, b)
```

Тобто максимізація cosine similarity або dot product автоматично еквівалентна мінімізації L2-distance. Саме тому в даному експерименті всі три метрики повернули однаковий top-5.

### Що було б, якби ембеддинги не були нормалізовані

Якщо ембеддинги не нормалізувати, ситуація зміниться. У цьому випадку dot product почне залежати не тільки від напрямку векторів, а й від їхньої довжини. Документи з більшими за модулем векторами можуть отримувати вищі оцінки навіть тоді, коли їхній зміст менш релевантний запиту. Cosine similarity усуває цей ефект шляхом ділення на довжини векторів і порівнює лише напрямки. L2-distance також починає враховувати абсолютні масштаби векторів. Тому без нормалізації результати cosine similarity, dot product та L2-distance зазвичай розходяться, а top-5 документів можуть суттєво відрізнятися.

## 7. Частина 5 — Chunking

Скрипт `scripts/05_chunking.py`:

- вибирає 30 статей з найдовшими abstract;
- створює чанки двома способами:
  - fixed-size chunking;
  - sentence-preserving semantic chunking;
- створює два окремі Pinecone-індекси:
  - `arxiv-chunks-fixed`;
  - `arxiv-chunks-semantic`;
- генерує ембеддинги для кожного чанка;
- завантажує чанки в Pinecone;
- виконує пошук по кількох тестових запитах.

Запуск:

```bash
python scripts/05_chunking.py
```

У моєму запуску було створено:

```text
Fixed-size chunks: 94
Sentence-preserving chunks: 92
```

Тобто обидві стратегії дали близьку кількість чанків, але sentence-preserving chunking створив трохи менше чанків, оскільки обʼєднував речення у більш цілісні блоки.

### Яка стратегія дає більш осмислені чанки

Більш осмислені чанки зазвичай дає sentence-preserving chunking. Причина проста: він не розрізає текст посеред речення і зберігає завершені смислові блоки. Fixed-size chunking простіший і стабільніший за розміром, але може механічно обрізати речення, через що embedding чанка містить неповну думку.

У моєму експерименті різниця була помітна у самих текстах чанків. Для fixed-size chunking деякі чанки починалися або закінчувалися посеред думки, наприклад:

the final masses in the different regions and the locations of the boundaries...

Такий chunk технічно індексується, але він менш зрозумілий як окремий фрагмент тексту. У sentence-preserving chunking фрагменти частіше виглядали як завершені уривки, наприклад:

We propose an evolutionary sequence for massive galaxies that smoothly connects normal star-forming galaxies to quiescent galaxies...

Тому з погляду читабельності та якості контексту для RAG sentence-preserving chunking є кращим.

Водночас результати пошуку для тестових запитів не були ідеальними. Наприклад, для запитів `neural networks for image recognition`, `reinforcement learning for decision making` і `language models for text generation` у top-5 часто потрапляли статті з астрофізики, квантової фізики або статистичної механіки. Це означає, що на маленькому піднаборі з 30 довгих abstract не обовʼязково є релевантні документи під ці запити. Тому якість результатів у цьому експерименті більше обмежена складом тестового датасету, ніж самим підходом до chunking.

### Чи є випадки розрізаних речень і як це впливає на embedding

У fixed-size chunking такі випадки можливі, бо текст ріжеться за кількістю слів, а не за межами речень. Це може погіршити embedding, бо модель отримує неповний або обірваний контекст. Наприклад, якщо важливе означення почалося в одному чанку, а закінчилося в іншому, кожен окремий embedding буде менш точним.

У результатах це видно на прикладах, де fixed-size chunk починається не з повного речення, а з продовження попереднього фрагмента:

degree of helicity...

або:

the final masses in the different regions...

Такі фрагменти складніше інтерпретувати окремо, бо частина контексту залишилася в попередньому чанку. У sentence-preserving chunking ця проблема менша, бо chunk формується з цілих речень. Мінус у тому, що розмір чанків може бути менш рівномірним.

### Як overlap впливає на кількість чанків і покриття тексту

Overlap збільшує кількість чанків, але покращує покриття контексту. Якщо overlap = 0, кожне слово потрапляє тільки в один chunk. Якщо overlap > 0, частина слів дублюється в сусідніх чанках. Це корисно, бо інформація біля межі чанка не губиться.

У цьому завданні fixed-size chunking використовує overlap, тому він створив трохи більше чанків:

94 fixed-size chunks проти 92 sentence-preserving chunks

Перевага overlap у тому, що він зменшує ризик втратити зміст на межі двох чанків. Недолік — збільшення витрат: більше чанків, більше ембеддингів, більше записів у Pinecone і більше storage.


## 8. Частина 6 — Гібридний пошук

Скрипт `scripts/06_hybrid_search.py` реалізує три типи пошуку:

1. BM25;
2. векторний пошук у Pinecone;
3. гібридний пошук через Reciprocal Rank Fusion.

Запуск:

```bash
python scripts/06_hybrid_search.py
```

Тестові запити:

```text
BERT fine-tuning
Yann LeCun convolutional networks
making computers understand human emotions from text
```

### Як працює RRF

Reciprocal Rank Fusion обʼєднує кілька ранжованих списків. Для кожного документа рахується score:

```text
RRF_score(d) = sum(1 / (k + rank_i(d)))
```

Якщо документ високо стоїть у BM25 і також високо стоїть у vector search, він отримає більший сумарний score. Якщо документ є тільки в одному списку, він теж може потрапити у фінальний top-K, але його score буде нижчим.

У моєму запуску RRF не завжди давав ідеально релевантний результат, але добре показав сам механізм fusion: фінальна видача комбінувала документи з BM25 і vector search, а не просто копіювала один із методів.

### Який метод дав кращий результат і чому

За фактичними результатами запуску найстабільніше виглядав BM25 для запитів із точними словами. Наприклад, для запиту `BERT fine-tuning` BM25 вивів статті з буквальним збігом `fine tuning`:

```text
A New Measure of Fine Tuning
The NMSSM Solution to the Fine-Tuning Problem
Fine-Tuning in Brane-antibrane Inflation
```

Хоча ці статті не про BERT, BM25 коректно відпрацював як lexical search: він знайшов документи з точним терміном `fine-tuning`. Це показує як сильну, так і слабку сторону BM25: він добре ловить буквальні збіги, але не розуміє, що в запиті очікується саме ML/NLP-контекст.

Для запиту `Yann LeCun convolutional networks` BM25 також відреагував на точні слова, особливо `convolutional` і `networks`, але результати були змішані: частина документів стосувалася convolutional codes або complex networks, а не нейронних convolutional networks. Це приклад lexical ambiguity: слова збігаються, але домен може бути іншим.

Vector search у цьому експерименті не завжди був кращим. Для `BERT fine-tuning` він повернув нерелевантні або слабко повʼязані документи, наприклад `Misere quotients for impartial games` і `Abstract Convexity and Cone-Vexing Abstractions`. Для `Yann LeCun convolutional networks` vector search знайшов деякі ближчі за змістом результати, наприклад `Multilayer Perceptron with Functional Inputs` і `Adaptive classification of temporal signals in fixed-weights recurrent neural networks`, але все одно не знайшов явно релевантні роботи про LeCun або CNN.

Найкращий окремий приклад для hybrid search був запит `making computers understand human emotions from text`. BM25 поставив на перше місце `On the Development of Text Input Method - Lessons Learned`, і vector search також включив цей документ у top-5. Через це RRF підняв його на перше місце у hybrid-видачі. Це логічно: документ одночасно мав lexical-сигнал за словом `text` і semantic-сигнал, повʼязаний із computer processing of language.

Загальний висновок: у цьому конкретному запуску BM25 краще працював для точних термінів, vector search давав ширші семантичні збіги, а hybrid RRF був корисним як компроміс. Водночас якість результатів обмежена самим піднабором arXiv: більшість записів у прикладах належить до 2007 року, і в маленькому датасеті може просто не бути справді релевантних статей про BERT, LeCun або emotion recognition.

### Чи є документи в top-5 hybrid, яких немає в top-5 окремих методів

У наведеному запуску top-5 hybrid переважно складався з документів, які вже були присутні або в BM25 top-5, або у vector top-5. Тобто в цих конкретних результатах немає виразного прикладу документа, який не входив у top-5 жодного окремого методу, але потрапив у top-5 hybrid.

Однак теоретично така ситуація можлива, якщо RRF обʼєднує ширші списки, наприклад top-10 або top-50. Документ може бути на 6-му місці в BM25 і на 6-му місці у vector search. Окремо він не входить у top-5 жодного методу, але після fusion отримує достатній сумарний score, щоб піднятися у фінальний top-5.

У моєму експерименті видно інший важливий ефект RRF: якщо документ присутній в обох списках, він може отримати сильне підсилення. Наприклад, для запиту `making computers understand human emotions from text` документ `On the Development of Text Input Method - Lessons Learned` був #1 у BM25 і #2 у vector search, тому став #1 у hybrid RRF.

### Як зміна параметра `k` в RRF впливає на видачу

Параметр `k` у RRF контролює, наскільки сильно враховується різниця між позиціями в ранжованих списках.

Якщо `k` малий, наприклад `k=1`, перші позиції отримують дуже сильну перевагу. Різниця між rank 1 і rank 5 стає великою, тому один метод може агресивно впливати на фінальний результат.

Якщо `k` більший, наприклад `k=60`, різниця між сусідніми позиціями згладжується. У такому випадку RRF стає стабільнішим і менше залежить від випадкового першого місця в одному з методів. Саме тому в пошукових системах часто використовують значення близько `k=60`: воно дає більш обережне обʼєднання BM25 і vector search.

У моєму результаті RRF-score має значення приблизно `0.016–0.032`, що відповідає логіці формули з великим `k`: окремий rank 1 дає приблизно `1 / (60 + 1) = 0.01639`, а документ, який є високо в обох списках, може отримати майже подвійну суму. Саме тому `On the Development of Text Input Method - Lessons Learned` для третього запиту отримав RRF-score `0.032522`: він був одночасно #1 у BM25 і #2 у vector search.

## 9. Розгорнутий аналіз компромісів

### 9.1. Семантичний пошук vs BM25

BM25 краще працює для точних ключових слів, назв методів, абревіатур, імен авторів і термінів, які повинні буквально зустрічатися в документі. У моїх результатах це добре видно на запиті `BERT fine-tuning`: BM25 знайшов документи з точним терміном `fine tuning`, хоча вони були з фізики, а не з NLP. Тобто BM25 сильний у буквальному збігу, але не завжди правильно розуміє предметну область.

Семантичний пошук краще підходить для перефразованих запитів, коли релевантний документ може не містити тих самих слів, що й запит. Наприклад, для запиту `making computers understand human emotions from text` vector search підняв документи про opinion dynamics, collective attention, narratives та text input. Частина результатів не була ідеальною, але вони показують, що vector search шукає ширшу змістову близькість, а не тільки точні слова.

Загальне правило: якщо важливі точні терміни — BM25; якщо важливий зміст і перефразування — semantic search; якщо потрібна стабільніша якість — hybrid search. Hybrid RRF корисний саме тому, що може підсилити документи, які мають і lexical, і semantic сигнал.

### 9.2. Вплив розміру чанка

Якщо chunk занадто маленький, наприклад 10–15 слів, він може не містити достатньо контексту. Embedding такого чанка буде нестабільним, бо модель бачить лише фрагмент думки. Це може призвести до випадкових або поверхневих збігів.

Якщо chunk занадто великий, наприклад 500+ слів, у ньому може бути кілька різних тем. Embedding стає усередненим представленням багатьох змістів, і конкретний запит може погано збігатися з таким великим блоком. Крім того, багато sentence-transformer моделей мають обмеження по кількості токенів, тому надто довгий текст може обрізатися.

Оптимальний розмір залежить від задачі. Для коротких abstract можна працювати з цілим abstract. Для PDF, книг або звітів краще використовувати чанки приблизно 100–300 слів з overlap, щоб зберегти локальний контекст і не втратити інформацію на межах.

### 9.3. Невідповідна метрика: euclidean vs cosine

Якщо індекс Pinecone створити з метрикою `euclidean`, але ембеддинги використовувати як cosine-based semantic embeddings, це може змінити ранжування, особливо якщо вектори не нормалізовані. Для одиничних векторів L2 і cosine математично повʼязані:

```text
||a - b||² = ||a||² + ||b||² - 2 * dot(a, b)
```

Якщо `||a|| = ||b|| = 1`, тоді:

```text
||a - b||² = 2 - 2 * dot(a, b)
```

А оскільки для нормалізованих векторів `dot(a, b) = cosine(a, b)`, мінімізація L2 еквівалентна максимізації cosine. Але якщо вектори не нормалізовані, ця еквівалентність зникає. Тому для normalized sentence-transformer embeddings найбезпечніше використовувати `cosine` і явно нормалізувати ембеддинги перед завантаженням.

### 9.4. Обмеження Pinecone Starter

Для навчальної роботи з 10 000 статей Pinecone Starter/serverless достатній. Потенційні обмеження: ліміти на кількість записів, write/read units, розмір metadata, кількість індексів і вартість при масштабуванні. У цій роботі abstract обрізається в metadata, щоб не перевищувати ліміти й не зберігати зайвий текст у векторній базі.

Якби датасет був не 10 000, а 10 мільйонів статей, я б змінив архітектуру. По-перше, повні тексти зберігав би окремо в object storage або документній базі, а у Pinecone — тільки vector id і коротку metadata. По-друге, завантаження робив би через batch pipeline з retry, логуванням і контролем помилок. По-третє, обовʼязково додав би offline evaluation: recall@K, MRR, порівняння BM25/vector/hybrid на тестовому наборі запитів. Також варто розглянути paid tier або альтернативи на кшталт Qdrant/Milvus/Elasticsearch/OpenSearch, залежно від бюджету, latency, filtering requirements і вимог до self-hosting.
